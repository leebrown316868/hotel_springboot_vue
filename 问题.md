# 测试
正确的调试方法应该是
  1. 第一时间查看浏览器Console的所有错误信息
  2. 检查组件的基本导入是否正确
  3. 使用无痕模式或清除缓存避免缓存问题
  4. 不要过度修改代码，专注于实际问题
 教训
  - 简单问题简单解决，不要过度复杂化
  - 先看错误信息，再进行调试
  - 清除缓存是解决前端问题的有效方法
  - Firefox的调试输出比Chrome更清晰
# 登录
管理平台不要注册功能，注册在登陆后台的管理系统内
保存登录状态
前端请求的是 /auth/login，但后端配置的是 /api/auth/login，注意前后端一致
问题1：登录页面"闪烁"无法进入系统 - Vue Router 4导航守卫使用返回值而不是next()回调，导致导航异常。解决：使用返回值语法：`router.beforeEach((to) => { ... return { name: 'login' } })`，不返回任何值表示允许导航
问题2：/api/auth/me返回500空指针异常 - AuthController.getCurrentUser()没有处理userDetails为null的情况。解决：添加null检查返回401状态码。
问题3：登录返回401 Invalid email or password - 旧数据库密码编码问题。解决：删除data/hotel.db文件，重启后端重新初始化数据库。
问题4：登录页面输入错误密码时页面刷新 - api.ts响应拦截器在收到401错误时会执行window.location.href重定向，导致登录页面刷新。解决：只在非登录页面时才处理401错误并重定向，登录页面的401错误由Login.vue组件自己处理。

# 管理员模块
## 系统管理
问题1：Vue Router警告"The next() callback in navigation guards is deprecated" - 使用了旧的next()回调语法。解决：移除next参数，使用返回值：需要重定向时返回路由对象，允许导航时不返回任何值。

问题2：async-validator警告"confirmPassword is required" - ChangePasswordDialog中confirmPassword不在form对象中，导致验证规则无法正常工作。解决：将confirmPassword作为临时字段加入form对象，提交前解构排除该字段。

# Maven编译问题
Maven编译失败，annotationProcessorPath dependencies version为空 - maven-compiler-plugin的annotationProcessorPaths中Lombok依赖缺少<version>标签。解决：在<properties>添加<lombok.version>1.18.24</lombok.version>，并在Lombok path中添加<version>${lombok.version}</version>。同时修复RoomTypeAlreadyExistsException和RoomTypeInUseException重复构造函数问题，删除RoomService中不存在的RemoteRequest导入，在RoomRepository添加countByType方法。

# 房型配置同步问题
问题1：rooms-resource页面编辑房型配置成功但数据库未更新 - 前端读取房型从room_types表，但更新调用/api/settings/room-types更新system_settings表的JSON字段，两个数据源不同步。解决：修改前端使用roomType API的updateRoomType方法直接更新room_types表。

问题2：rooms页面房间信息与数据库不同步 - RoomMapper.toResponse()返回枚举的displayName（中文名称），前端期望枚举名称（如SINGLE）来查找房型配置。解决：修改RoomMapper使用.name()代替.getDisplayName()返回枚举名称。

问题2：导航栏没有根据角色显示不同模块 - Layout.vue的menuItems是硬编码的，显示所有模块给所有用户。解决：将menuItems改为computed属性，根据用户角色过滤显示对应模块（STAFF→前台员工模块，CUSTOMER→用户模块，ADMIN→所有模块）。

问题3：rooms-resource页面修改房型配置后rooms页面不同步 - rooms页面没有监听房型配置更新事件。解决：在RoomsResource.vue保存房型配置后发送localStorage事件，在Rooms.vue监听storage事件并刷新房型配置。

问题4：房间容量没有从房型配置动态获取 - RoomMapper.toResponse()直接从rooms表获取冗余的capacity字段，导致房型配置更新时容量不变。解决：修改RoomMapper从room_types表动态获取capacity字段。

问题5：房间价格与房型配置不同步 - 更新房型配置的base_price时没有同步更新rooms表的price字段。解决：在RoomTypeServiceImpl.updateRoomType()中检测价格变化后调用roomRepository.updatePriceByType()批量更新所有关联房间。

问题6：updatePriceByType方法返回500错误"Query does not return results" - UPDATE语句缺少@Modifying注解，Spring Data JPA将其当作查询处理导致返回结果集错误。解决：添加@Modifying和@Transactional注解标识修改操作。

# 搜索可用房间问题
问题1：多房型选择时报错"HQL语法错误：unexpected AST node: {vector}" - 前端发送数组参数格式roomTypes[]=DOUBLE&roomTypes[]=EXECUTIVE_SUITE导致Spring无法正确绑定为List<RoomType>。解决：前端使用自定义paramsSerializer将数组转为逗号分隔字符串roomTypesStr=DOUBLE,EXECUTIVE_SUITE，后端RoomSearchRequest添加getRoomTypes()方法将字符串转换为枚举列表。

问题2：入住人数筛选逻辑不对 - 查询条件r.capacity >= :guestCount使用rooms表的capacity字段，但房型配置修改时没有同步更新rooms表的容量。解决：在RoomTypeServiceImpl.updateRoomType()中检测容量变化后调用roomRepository.updateCapacityByType()批量更新所有关联房间的容量。

问题3：多房型选择JPQL查询500错误"unexpected AST node: {vector}" - JPQL查询中`(:roomTypes IS NULL OR r.type IN :roomTypes)`条件导致Hibernate将列表参数错误展开为`roomTypes_0, roomTypes_1`，生成语法错误的SQL。解决：将查询拆分为两个方法，findAvailableRoomsNoTypeFilter()用于无房型筛选，findAvailableRoomsWithTypeFilter()用于有房型筛选，在BookingServiceImpl根据roomTypes是否为空调用不同方法。

问题4：后端启动失败"NOT NULL constraint failed: rooms.capacity" - Room实体使用@Builder和@AllArgsConstructor时，Lombok生成的构造函数包含所有字段（包括createdAt和updatedAt），与@PrePersist生命周期方法冲突，导致capacity字段未被正确设置。解决：移除@Builder和@AllArgsConstructor注解，保留@Data和@NoArgsConstructor，在DataInitializer.createRoom()中使用new Room() + setter方式创建对象。

问题5：StaffBookings页面房型显示code而不是name，Dashboard最近预定未排序 - StaffBookings.vue的getRoomTypeName映射表不完整缺少EXECUTIVE_SUITE和PRESIDENTIAL_SUITE，Dashboard的StatisticsServiceImpl.getRecentBookings()查询未按创建时间倒序。解决：更新前端房型映射表添加完整枚举，后端查询添加Sort.by(Sort.Direction.DESC, "createdAt")排序。

问题6：预订时客人信息未同步到guests表 - 前端BookingWizard收集了客人信息但未发送给后端，后端只接受guestId假设Guest已存在。解决：BookingRequest添加GuestInfo内部类，BookingServiceImpl添加findOrCreateGuest()方法根据email查找或创建Guest，前端将guestForm数据作为guestInfo发送。

# 公开页面访问权限问题
问题1：未登录用户访问房间浏览页面时，`/api/room-types/active` 返回403错误 - SecurityConfig中缺少公开访问规则。解决：添加`.antMatchers("/api/room-types/active").permitAll()`。

问题2：未登录用户搜索可用房间时，`/api/bookings/available-rooms` 返回403错误 - SecurityConfig中缺少公开访问规则。解决：添加`.antMatchers("/api/bookings/available-rooms").permitAll()`。

问题3：STAFF用户编辑房间时返回500 Access Denied - SecurityConfig中的`.antMatchers("/api/rooms/*").permitAll()`太宽泛，允许所有HTTP方法访问房间API，与方法级别的@PreAuthorize("hasRole('ADMIN')")注解冲突。解决：改为`.antMatchers(HttpMethod.GET, "/api/rooms/*").permitAll()`只允许GET请求公开访问（未登录用户查看房间详情），PUT/POST/DELETE仍需认证。

# 房间图片上传功能
实现Rooms.vue编辑对话框中的图片上传功能，上传的图片同步到BrowseRooms.vue和RoomDetail.vue展示。

后端实现：
- 创建FileUploadConfig配置静态资源访问，映射/uploads/**到本地uploads目录
- RoomController新增POST /api/rooms/{id}/images接口处理图片上传
- RoomServiceImpl.uploadRoomImage实现文件保存逻辑：
  - 目录结构：uploads/rooms/{roomId}/
  - 文件命名：{timestamp}_{random}.{ext}
  - 支持格式：JPG、PNG、WEBP
  - 文件大小限制：5MB
- RoomMapper修复toEntity方法，移除builder模式（Room类无@Builder注解）

前端实现：
- Rooms.vue添加el-upload图片上传组件（list-type="picture-card"）
- 支持最多5张图片，预览、删除功能
- 编辑房间时加载现有图片
- 提交时将图片URL数组保存到room.images字段（JSON格式）
- room.ts API添加uploadRoomImage方法

问题1：图片上传404错误 - vite.config.ts缺少API代理配置，前端请求/api/rooms/1/images发送到localhost:3000而不是后端8080。解决：添加proxy配置将/api请求转发到http://localhost:8080。

问题2：图片访问CORS错误 - 访问/uploads/rooms/1/xxx.webp返回NS_BINDING_ABORTED和OpaqueResponseBlocking错误。解决：SecurityConfig添加.antMatchers("/uploads/**").permitAll()允许公开访问静态资源，FileUploadConfig添加CORS配置允许localhost:3000跨域访问。

问题3：图片没有同步到BrowseRooms - RoomRequest和RoomResponse DTO缺少images字段，导致前端提交的images数据无法保存到数据库，API返回也不包含images字段。解决：
- RoomRequest添加images字段
- RoomResponse添加images字段
- RoomMapper.toEntity添加room.setImages()
- RoomMapper.toResponse添加.images(room.getImages())
- 前端types/room.ts的RoomRequest和RoomResponse添加images字段

问题4：RoomDetail页面图片无法加载 - roomImages解析的图片URL是相对路径/uploads/rooms/1/xxx.webp，浏览器无法正确加载。解决：在roomImages computed中添加URL转换逻辑，将相对路径转换为http://localhost:8080/uploads/rooms/1/xxx.webp。

问题5：数据库images字段为空 - 用户上传图片后没有点击"确定"按钮保存，图片只保存到文件系统但没有写入数据库。解决：指导用户完整操作流程（上传图片→点击确定→保存），或直接用Python更新数据库添加images数据用于测试。
