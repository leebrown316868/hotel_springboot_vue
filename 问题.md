# 房型配置同步问题

# 搜索可用房间问题
问题1：多房型选择时报错"HQL语法错误：unexpected AST node: {vector}" - 前端发送数组参数格式roomTypes[]=DOUBLE&roomTypes[]=EXECUTIVE_SUITE导致Spring无法正确绑定为List<RoomType>。解决：前端使用自定义paramsSerializer将数组转为逗号分隔字符串roomTypesStr=DOUBLE,EXECUTIVE_SUITE，后端RoomSearchRequest添加getRoomTypes()方法将字符串转换为枚举列表。

问题2：入住人数筛选逻辑不对 - 查询条件r.capacity >= :guestCount使用rooms表的capacity字段，但房型配置修改时没有同步更新rooms表的容量。解决：在RoomTypeServiceImpl.updateRoomType()中检测容量变化后调用roomRepository.updateCapacityByType()批量更新所有关联房间的容量。

问题3：多房型选择JPQL查询500错误"unexpected AST node: {vector}" - JPQL查询中`(:roomTypes IS NULL OR r.type IN :roomTypes)`条件导致Hibernate将列表参数错误展开为`roomTypes_0, roomTypes_1`，生成语法错误的SQL。解决：将查询拆分为两个方法，findAvailableRoomsNoTypeFilter()用于无房型筛选，findAvailableRoomsWithTypeFilter()用于有房型筛选，在BookingServiceImpl根据roomTypes是否为空调用不同方法。

问题4：后端启动失败"NOT NULL constraint failed: rooms.capacity" - Room实体使用@Builder和@AllArgsConstructor时，Lombok生成的构造函数包含所有字段（包括createdAt和updatedAt），与@PrePersist生命周期方法冲突，导致capacity字段未被正确设置。解决：移除@Builder和@AllArgsConstructor注解，保留@Data和@NoArgsConstructor，在DataInitializer.createRoom()中使用new Room() + setter方式创建对象。

问题5：StaffBookings页面房型显示code而不是name，Dashboard最近预定未排序 - StaffBookings.vue的getRoomTypeName映射表不完整缺少EXECUTIVE_SUITE和PRESIDENTIAL_SUITE，Dashboard的StatisticsServiceImpl.getRecentBookings()查询未按创建时间倒序。解决：更新前端房型映射表添加完整枚举，后端查询添加Sort.by(Sort.Direction.DESC, "createdAt")排序。

问题6：预订时客人信息未同步到guests表 - 前端BookingWizard收集了客人信息但未发送给后端，后端只接受guestId假设Guest已存在。解决：BookingRequest添加GuestInfo内部类，BookingServiceImpl添加findOrCreateGuest()方法根据email查找或创建Guest，前端将guestForm数据作为guestInfo发送。

# 前端白屏问题（2026-03-18）
实现日期: 2026-03-18

## 问题描述
更新用户注册自动填充功能后，前端界面显示完全空白，浏览器控制台无错误输出。

## 根本原因
多个 TypeScript 编译错误和路由守卫配置问题导致前端无法正常编译和运行。

## 修复的问题列表

### 1. 路由守卫参数错误
**文件:** `frontend/src/router/index.ts:116`
**问题:** 路由守卫使用了 Vue Router 4 的返回值语法，但还保留了旧的 `next` 参数
**错误信息:** `The "next" callback was never called inside of... If you are returning a value instead of calling "next", make sure to remove the "next" parameter from your function.`
**修复:** 移除 `next` 参数，只保留 `(to, from)`
```javascript
// 修复前
router.beforeEach((to, from, next) => { ... })

// 修复后
router.beforeEach((to, from) => { ... })
```

### 2. Login.vue 缺少图标导入
**文件:** `frontend/src/views/Login.vue:5`
**问题:** 使用了 `OfficeBuilding` 图标组件但未导入
**修复:** 添加 `import { OfficeBuilding } from '@element-plus/icons-vue'`

### 3. auth.ts 缺少类型导入
**文件:** `frontend/src/utils/auth.ts:2`
**问题:** 使用了 `ApiResponse` 类型但未导入，导致 TypeScript 编译错误
**修复:** 添加 `ApiResponse` 到导入列表
```typescript
// 修复前
import type { AuthRequest, AuthResponse, RegisterRequest, User } from '../types/auth'

// 修复后
import type { AuthRequest, AuthResponse, RegisterRequest, User, ApiResponse } from '../types/auth'
```

### 4. tsconfig.json 路径别名配置错误
**文件:** `frontend/tsconfig.json:22`
**问题:** TypeScript 路径别名配置为 `"./*"` 而不是 `"./src/*"`，导致无法找到 `@/` 开头的模块
**修复:** 修改 paths 配置
```json
// 修复前
"paths": {
  "@/*": ["./*"]
}

// 修复后
"paths": {
  "@/*": ["./src/*"]
}
```

### 5. roomType.ts 类型导入路径错误
**文件:** `frontend/src/api/roomType.ts:3`
**问题:** 从错误的路径导入 `ApiResponse` 类型
**修复:** 从 `@/types/api` 导入 `ApiResponse`
```typescript
// 修复前
import type { ApiResponse, RoomTypeListResponse, ... } from '@/types/roomType'

// 修复后
import type { ApiResponse } from '@/types/api'
import type { RoomTypeListResponse, ... } from '@/types/roomType'
```

# HTTP 500 错误：搜索带房型筛选的可用房间失败
日期: 2026-03-20

## 问题描述
前端带着指定房型参数（例如 `roomTypesStr=EXECUTIVE_SUITE,DOUBLE`）请求 `/api/bookings/available-rooms` 接口时，服务端直接抛出 `HTTP 500 内部服务器错误`。

## 根本原因
1. **参数类型错配**：`BookingRepository` 的 JPQL `@Query` 接口 `findAvailableRoomsWithTypeFilter` 原本接收的 `checkIn` 和 `checkOut` 是 `String` 类型，但在底层 `Booking` 实体中时间字段是 `LocalDate` 类型。Spring Data JPA 无法在 JPQL 里自动把 String 和 LocalDate 相比较。
2. **JPQL 对 Enum 字符串硬关联解析错误**：查询条件中的 `b.status NOT IN ('CANCELLED', 'CHECKED_OUT')` 试图在 JPQL 直接用字符串与枚举类型的 `status` 字段进行比对，导致 Hibernate 拦截并报类型转换异常。
3. **IN 子句 Enum 映射错误**：查询条件中 `r.type IN :roomTypes` 的 `:roomTypes` 接收的是 `List<String>`，而 `Room.type` 是 `RoomType` 枚举，这同样引发类型转化报错。

## 修复方式
### 1. 修正 Repository 接口参数类型
将 `BookingRepository.java` 中的相关入参全部还原并转配为 `LocalDate`。同时，去除写死的 `'CANCELLED'` 字符串，改为在 Java 层接收并传入 `List<BookingStatus> excludeStatuses` 排除状态。把入参 `@Param("roomTypes")` 的泛型强转为 `List<RoomType>` 解决映射错配。

### 2. 在 Service 层转好枚举参数
在 `BookingServiceImpl.java` 中拿到前端传进来的房型字符串列表后，使用 Stream 和 `RoomType::valueOf` 映射成原生的枚举集合 `List<RoomType>`，最后将它与准备好的 `excludeStatuses` 枚举列表一同传给 Repository 调用方法，使占位符与 DB 实体模型完美贴合。

## 经验教训
1. **优先确保对齐 JPA 的枚举（Enum）及时间类型**：编写通过 `@Query` 注解构建的自定义 JPQL 语句时，传给 Hibernate 的占位参数最好与相关实体类的原始属性类型保持绝对一致，尽量避免在字串内生硬编排未脱敏的内容字面量比对。
2. **500 内部错误追溯**：出现模糊未脱逃的 HTTP 500 控制器响应时，应当定位到项目本身跑起来的 stdout 输出或是 log 去查看具体错误栈，快速确认是 Spring 验证器、JSON转化或是底层 JPA 数据流转哪里抛出的底错。

# 取消预订引发前端 10s 超时报错及页面静默挂起
日期: 2026-03-20

## 问题描述
用户在【我的订单】页面点击“取消预订”时，接口长时间不响应，随后抛出 HTTP 10000ms Timeout 超时错误，报错堆栈却离奇地指向了 `/api/settings/public`。

## 根本原因
1. **认证对象向下强转引发空指针（500 错误）**：`BookingController` 中的 `cancelBooking` 路由入口不慎硬编码了 `@AuthenticationPrincipal UserDetailsImpl userDetails`。以普通系统客户（Guest）状态发起调用时，其身份实例为 `GuestDetailsImpl`。在 Spring 默认机制下，转换失败导致给其自动注入了 `null`。致使提取标志位时的 `.getId()` 直接发生 `NullPointerException` 崩溃。
2. **引发前端拦截器雪崩跳板**：由于全局的 Axios Error 拦截器接收到未捕获明确消息的异状，强制触发登录失效跳转逻辑去往 `/login` 页，而此时 Login 又在试图并发去拉取酒店系统全局配置 `/api/settings/public` 信息。
3. **PowerShell 机制性挂起后台**：与此同时，如果在运行起后端的 Windows 控制台中发生过鼠标点按等操作，极易误触其独有的“快速编辑模式”。这会将整个底层 Java 支撑程序的 Stdout 输出流阻塞，连带掐断了 Tomcat 处理剩余 Socket 监听的能力并整体挂起，也就是在这时，那笔 `settings` 的请求死锁在发送途中直到超出了浏览器自身限制的 10 秒上限。

## 修复方式
1. 放宽接口接收的主体签名，改为高度兼容的基础接口 `@AuthenticationPrincipal org.springframework.security.core.userdetails.UserDetails userDetails`；
2. 方法体内部通过增加 `instanceof` 手工核查是内部员工、还是外部顾客的封装类，安全解构获得精确的 `userId`；
3. 将受到快速编辑模式挂起影响的当前后台服务作清理，并执行 `mvn spring-boot:run` 使新控制层载入应用。

# 历史订单提交评价成功后无提示引发复提交拦截
日期: 2026-03-20

## 问题描述
在 `历史记录与评价反馈` 组件编写评语且其实已被云端接纳收录时，却并未弹出操作成功的弹窗，也未发生视图状态关闭。直到重新点提交，被上游拦截返回了由于“重复评价”引发的报错提示。

## 根本原因
前端 `HistoryFeedback.vue` 提取回调时，以 `if (response.data.code === 201)` 作为了渲染成功分支的卡点。但现实情况是，后端 `ReviewController` 的底层返回 HTTP Status 虽然挂载着标准的被创建 201，但其实际回传的标准化 `ApiResponse.success()` 载体内部写死的业务识别码 JSON 载荷 `code` 仍保留于 `200`，造成了永不可能成立的前端拦截判定，令成功后的处理被完全静默吞没。

## 修复方式
打补丁兼容后端接口载体内与外两套状态码。修改 Vue 判断门槛为组合判断逻辑：`if (response.data.code === 200 || response.status === 201)`，彻底恢复正确分支导向。
